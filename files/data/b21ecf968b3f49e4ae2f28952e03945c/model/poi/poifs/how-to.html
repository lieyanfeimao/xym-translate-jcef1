How To Use the POIFS APIs(${t_676b951700224970a3300721e15e933b})

This document describes how to use the POIFS APIs to read, write, and modify files that employ a POIFS-compatible data structure to organize their content.(${t_e62430880a8545f2a2cee6d42ded1c25})

Target Audience(${t_9daf677aa174407a8439b68036bdc678})

This document is intended for Java developers who need to use the POIFS APIs to read, write, or modify files that employ a POIFS-compatible data structure to organize their content. It is not necessary for developers to understand the POIFS data structures, and an explanation of those data structures is beyond the scope of this document. It is expected that the members of the target audience will understand the rudiments of a hierarchical file system, and familiarity with the event pattern employed by Java APIs such as AWT would be helpful.(${t_49da45dc20a1432a89a55faec096a4cc})

Glossary(${t_7111f31e8c6746c1933db50d90f4349a})

This document attempts to be consistent in its terminology, which is defined here:(${t_b74fe30aae0f45b28c51204cd2594fca})

Directory
A special file that may contain other directories and documents.
DirectoryEntry
Representation of a directory within another directory.
Document
A file containing data, such as word processing data or a spreadsheet workbook.
DocumentEntry
Representation of a document within a directory.
Entry
Representation of a file in a directory.
File
A named entity, managed and contained by the file system.
File System
The POIFS data structures, plus the contained directories and documents, which are maintained in a hierarchical directory structure.
Root Directory
The directory at the base of a file system. All file systems have a root directory. The POIFS APIs will not allow the root directory to be removed or renamed, but it can be accessed for the purpose of reading its contents or adding files (directories and documents) to it.

The different ways of working with POIFS(${t_afd456287ea74714980a59bba9ba87ba})

The POIFS API provides ways to read, modify and write files and streams that employ a POIFS-compatible data structure to organize their content. The following use cases are covered:(${t_35175b4749f74ba2bbc6110c0116f0d4})

  • Reading a File System(${t_6d0cfcd17b8a404c97f61fbd653d3f88})
  • Conventional Reading with POIFSFileSystem(${t_d515ae372ef44be9ba84426178bcfd20})
  • Event-Driven Reading(${t_8a2fd2599662489a98b44250e52290ab})
  • Writing a File System(${t_0a2a5a106d2044f8a03a9fc71a3b208b})
  • Modifying a File System(${t_b6b8f57aa0f14aff9031d09bb0af2809})

Reading a File System(${t_0cebec0e86d147f597dc61ab047cbcfe})

This section covers reading a file system. There are two ways to read a file system; these techniques are sketched out in the following table, and then explained in greater depth in the sections following the table.(${t_d4327a6562ba41bc8286069dc1330c1e})

Conventional Reading with POIFSFileSystem
  • Simpler API similar to reading a conventional file system.(${t_182306dcaabb4359937ecc868ff478f2})
  • Can read documents in any order.(${t_06f365cc2ac14dd4b825bc14f4d60ebb})
  • Well tested read and write support.(${t_7072e533f7974fc3948c51009dcfd1aa})
  • If created from an InputStream, all files are resident in memory. (If created from a File, only certain key structures are)(${t_422b82e2ec02420ca3432a55a954ca74})
Event-Driven Reading
  • Reduced footprint -- only the documents you care about are processed.(${t_375adb22a8e94750a6e95c4f55dc8f37})
  • Improved performance -- no time is wasted reading the documents you're not interested in.(${t_ad2a4aa2c7424c6c8d68dfb883b214b2})
  • More complicated API.(${t_9392337ec0d34ebe878e38d2642e6027})
  • Need to know in advance which documents you want to read.(${t_661cff71b5f44a84b6e03ade14ca1ed4})
  • No control over the order in which the documents are read.(${t_6fbd10d6974e4a73a596462140298ba3})
  • No way to go back and get additional documents except to re-read the file system, which may not be possible, e.g., if the file system is being read from an input stream that lacks random access support.(${t_95f8fdc357e741bab3e279b198476fbf})

Conventional Reading with POIFSFileSystem(${t_0c67ab05eb8d4217b05387d44e4ca215})

In this technique for reading, certain key structures are loaded into memory, and the entire directory tree can be walked by the application, reading specific documents at leisure.(${t_85387dfed49246dfaf0d607fe5f6b3c6})

If you create a POIFSFileSystem instance from a File, the memory footprint is very small. However, if you createa a POIFSFileSystem instance from an input stream, then the whole contents must be buffered into memory to allow random access. As such, you should budget on memory use of up to 20% of the file size when using a File, or up to 120% of the file size when using an InputStream.(${t_c37a16ac8e2b47c0a9b2bcf65ba830d5})

Preparation(${t_c4cc9ce1280d438faa02b148de924c64})

Before an application can read a file from the file system, the file system needs to be opened and core parts processed. This is done using the org.apache.poi.poifs.filesystem.POIFSFileSystem class. Once the file system has been loaded into memory, the application may need the root directory. The following code fragment will accomplish this preparation stage:(${t_b51f169f014341408ef8fb7b13075d54})

DirectoryEntry root = fs.getRoot();
} catch (IOException e) {
// an I/O error occurred, or the File did not provide a compatible
// POIFS data structure
${t_166f4766df724f8391628da438b56a0c}
}
// Using an InputStream requires more memory than using a File
${t_9c967ee54dd14b6aa1cf9d11e85dc8b2}
try (POIFSFileSystem fs = new POIFSFileSystem(inputStream)) {
DirectoryEntry root = fs.getRoot();
} catch (IOException e) {
// an I/O error occurred, or the InputStream did not provide
// a compatible POIFS data structure
${t_d61f9b22049747d5acdb056d9b870191}

Reading a Specific Document(${t_f54b7d42bb594d228e94817e4e900c29})

There are a couple of ways to read a document, depending on whether the document resides in the root directory or in another directory. Either way, you will obtain an org.apache.poi.poifs.filesystem.DocumentInputStream instance.(${t_e33c469b807a4940806baa9b9969a53c})

DocumentInputStream

The DocumentInputStream class is a simple implementation of InputStream that makes a few guarantees worth noting:(${t_634bcc3270ce496f875561ef6284997a})

  • available() always returns the number of bytes in the document from your current position in the document.(${t_d9c69d8b9e9a4a8082c3a09bfd0af386})
  • markSupported() returns true.(${t_da6ecbbc66004a358f43612c66aa6e7b})
  • mark(int limit) ignores the limit parameter; basically the method marks the current position in the document.(${t_001cd8ef23734ff49bbfb4f19eef73b1})
  • reset() takes you back to the position when mark() was last called, or to the beginning of the document if mark() has not been called.(${t_4af81b47f4af427fb2a24f74d1653f92})
  • skip(long n) will take you to your current position + n (but not past the end of the document).(${t_ffd1f59d941349cb8fac1066cd6cb34c})

The behavior of available means you can read in a document in a single read call like this:(${t_ac6bd7dc08544d1a97e77bf702124ba5})

byte[] content = new byte[ stream.available() ];
stream.read(content);
stream.close();

The combination of mark, reset, and skip provide the basic mechanisms needed for random access of the document contents.(${t_dfc83483150447cfad10d64ee48c3bb5})

Reading a Document From the Root Directory

If the document resides in the root directory, you can obtain a DocumentInputStream like this:(${t_4aebc5c628f74f8faeb6582793590b7a})

// process data from stream
${t_5deabc240ba04985a1d511202b6640a7}
} catch (IOException e) {
// no such document, or the Entry represented by documentName is not a DocumentEntry
${t_5d6c4986754346be9cacd9a3d2a82fac}
}
Reading a Document From an Arbitrary Directory

A more generic technique for reading a document is to obtain an org.apache.poi.poifs.filesystem.DirectoryEntry instance for the directory containing the desired document (recall that you can use getRoot() to obtain the root directory from its file system). From that DirectoryEntry, you can then obtain a DocumentInputStream like this:(${t_c4ff0442a3114124a150e76b813a825e})

DocumentEntry document = (DocumentEntry)directory.getEntry(documentName);
DocumentInputStream stream = new DocumentInputStream(document);

Event-Driven Reading(${t_c7d7e5d862e7452baaf3e704c9107c1a})

The event-driven API for reading documents is a little more complicated and requires that your application know, in advance, which files it wants to read. The benefit of using this API is that each document is in memory just long enough for your application to read it, and documents that you never read at all are not in memory at all. When you're finished reading the documents you wanted, the file system has no data structures associated with it at all and can be discarded.(${t_c306885fb44c4142bd3f03e122b951e9})

POIFSReaderListener(${t_0990d7fcc90145a68698afdbe8b89511})

org.apache.poi.poifs.eventfilesystem.POIFSReaderListener is an interface used to register for documents. When a matching document is read by the org.apache.poi.poifs.eventfilesystem.POIFSReader, the POIFSReaderListener instance receives an org.apache.poi.poifs.eventfilesystem.POIFSReaderEvent instance, which contains an open DocumentInputStream and information about the document.(${t_29660ac0e864488ea09466945d621e36})

A POIFSReaderListener instance can register for individual documents, or it can register for all documents; once it has registered for all documents, subsequent (and previous!) registration requests for individual documents are ignored. There is no way to unregister a POIFSReaderListener.(${t_69af65cc60304d4ba4d9a43f839c679b})

Thus, it is possible to register a single POIFSReaderListener for multiple documents - one, some, or all documents. It is guaranteed that a single POIFSReaderListener will receive exactly one notification per registered document. There is no guarantee as to the order in which it will receive notification of its documents, as future implementations of POIFSReader are free to change the algorithm for walking the file system's directory structure.(${t_ca05590001ad424a97dcdbe36a34d38a})

It is also permitted to register more than one POIFSReaderListener for the same document. There is no guarantee of ordering for notification of POIFSReaderListener instances that have registered for the same document when POIFSReader processes that document.(${t_8ce89a05be9e4f66bf9b155b4ec1df38})

It is guaranteed that all notifications occur in the same thread. A future enhancement may be made to provide multi-threaded notifications, but such an enhancement would very probably be made in a new reader class, a ThreadedPOIFSReader perhaps.(${t_4dfc4c80257f4370809803c91d911c18})

The following describes the three ways to register a POIFSReaderListener for a document or set of documents:(${t_4d7575f8d60f4f96a9d68d67ed576d30})

registers listener for all documents.
registerListener(POIFSReaderListener listener)
registers listener for a document with the specified name in the root directory.
registerListener(POIFSReaderListener listener, String name)
registers listener for a document with the specified name in the directory described by path
registerListener(POIFSReaderListener listener, POIFSDocumentPath path, String name)

Processing POIFSReaderEvent Events(${t_86ae3bc2851b47a9a5c84b73ea84d42b})

Processing org.apache.poi.poifs.eventfilesystem.POIFSReaderEvent events is relatively easy. After all of the POIFSReaderListener instances have been registered with POIFSReader, the POIFSReader.read(InputStream stream) method is called.(${t_6ac6af256f95499799cdbd3f24b02af2})

Assuming that there are no problems with the data, as the POIFSReader processes the documents in the specified InputStream's data, it calls registered POIFSReaderListener instances' processPOIFSReaderEvent method with a POIFSReaderEvent instance.(${t_79fabe1fe83d4b96b3b4fb936d6cc748})

The POIFSReaderEvent instance contains information to identify the document (a POIFSDocumentPath object to identify the directory that the document is in, and the document name), and an open DocumentInputStream instance from which to read the document.(${t_adec2e1081f44a3a98f53b68bfe73646})

Writing a File System(${t_2f54419568fa4f779a429436b91fa697})

Writing a file system is very much like reading a file system in that there are multiple ways to do so. You can load an existing file system into memory and modify it (removing files, renaming files) and/or add new files to it, and write it, or you can start with a new, empty file system:(${t_ef40c3b5f2c24337aa46e0c818becb18})

POIFSFileSystem fs = new POIFSFileSystem();

The Naming of Names(${t_da4432741f7b4b97a7990f40d02dfd69})

There are two restrictions on the names of files in a file system that must be considered when creating files:(${t_8c114e3af1a440848a0e53f8427d5adf})

  1. The name of the file must not exceed 31 characters. If it does, the POIFS API will silently truncate the name to fit.(${t_ef20ec6a0fc94869a35c40d7b218576d})
  2. The name of the file must be unique within its containing directory. This seems pretty obvious, but if it isn't spelled out, there'll be hell to pay, to be sure. Uniqueness, of course, is determined after the name has been truncated, if the original name was too long to begin with.(${t_88eefee2cc294888a7b22120f8adf2f3})

Creating a Document(${t_3637c7dbd0c64c7d99bfeb921cd421c8})

A document can be created by acquiring a DirectoryEntry and calling one of the two createDocument methods:(${t_6eab30795a5d4b88b806dda8b6bf207f})

createDocument(String name, InputStream stream)
  • Simple API(${t_526fdab8bc714665818c3d973385702f})
  • Increased memory footprint (document is in memory until file system is written).(${t_27163db10ed5487dae78f6fc1ba8d4ec})
createDocument(String name, int size, POIFSWriterListener writer)
  • Decreased memory footprint (only very small documents are held in memory, and then only for a short time).(${t_c1fb656577134b78b414888bd895a9b9})
  • More complex API.(${t_9f8e135579cf4417a19489f13b117c12})
  • Determining document size in advance may be difficult.(${t_4496ea1219b64b1db366abc786b6a68c})
  • Lose control over when document is to be written.(${t_04272afbeaa0407ca8eb80fe65213e67})

Unlike reading, you don't have to choose between the in-memory and event-driven writing models; both can co-exist in the same file system.(${t_2f149217ccf3454785ef8b2a85504c27})

Writing is initiated when the POIFSFileSystem instance's writeFilesystem() method is called with an OutputStream to write to.(${t_9e1c4e7445944394ae7d0979472c5589})

The event-driven model is quite similar to the event-driven model for reading, in that the file system calls your org.apache.poi.poifs.filesystem.POIFSWriterListener when it's time to write your document, just as the POIFSReader calls your POIFSReaderListener when it's time to read your document. Internally, when writeFilesystem() is called, the final POIFS data structures are created and are written to the specified OutputStream. When the file system needs to write a document out that was created with the event-driven model, it calls the POIFSWriterListener back, calling its processPOIFSWriterEvent() method, passing an org.apache.poi.poifs.filesystem.POIFSWriterEvent instance. This object contains the POIFSDocumentPath and name of the document, its size, and an open org.apache.poi.poifs.filesystem.DocumentOutputStream to which to write. A DocumentOutputStream is a wrapper over the OutputStream that was provided to the POIFSFileSystem to write to, and has the responsibility of making sure that the document your application writes fits within the size you specified for it.(${t_466af6afc9974e30b614a1e76edd6e27})

If you are using a POIFSFileSystem loaded from a File with readOnly set to false, it is also possible to do an in-place write. Simply call writeFilesystem() to have the (limited) in-memory structures synced with the disk, then close() to finish.(${t_daa69d703fa44b6d8fe9843e732790d8})

Modifying a File System(${t_433a4900a2ce47a085d2490b46c4c431})

It is possible to modify an existing POIFS file system, whether it's one your application has loaded into memory, or one which you are creating on the fly.(${t_5be3d8d68c304249a1b173a6dde0dc84})

Removing a Document(${t_1781dc7ab6a549a3889ecff782b37332})

Removing a document is simple: you get the Entry corresponding to the document and call its delete() method. This is a boolean method, but should always return true, indicating that the operation succeeded.(${t_54381671f7a44f70b5d87b3650f2d17f})

Removing a Directory(${t_7f4a0ce6deef414a86c372650749fc1d})

Removing a directory is also simple: you get the Entry corresponding to the directory and call its delete() method. This is a boolean method, but, unlike deleting a document, may not always return true, indicating that the operation succeeded. Here are the reasons why the operation may fail:(${t_750293f6d98f49afbacaf8f70bfad4d7})

  • The directory still has files in it (to check, call isEmpty() on its DirectoryEntry; is the return value false?)(${t_21b28b9ff63849268ee8bd45bc9dd3ae})
  • The directory is the root directory. You cannot remove the root directory.(${t_4eb061508fbc43bfa678af6a3dc85ca8})

Changing a File's contents(${t_6f99b98618db40f29128b2def34bb312})

There are two ways available to change the contents of an existing file within a POIFS file system. One is using a DocumentOutputStream, the other is with POIFSDocument.replaceContents(${t_bf1ee9b88e994f049e882913b2b69cb6})

If you have available to you an InputStream to read the new File contents from, then the easiest way is via POIFSDocument.replaceContents. You would do something like:(${t_1dc7475a5abd4ad5b0db8e9f9d2fcadb})

${t_69f3829ca5db4e94a2cbe4ce317810a5}

DirectoryEntry root = fs.getRoot();
DocumentEntry myDocE = (DocumentEntry)root.getEntry("ToChange");

${t_a428f84cdf2d47e888d44260aea4fee7}

// Save the changes to the file in-place
${t_7ec6d515df16427ab91bad7a64214fa9}

fs.close();

Alternately, if you either have a byte array, or you need to write as you go along, then the OutputStream interface provided by DocumentOutputStream will likely be a better bet. Your code would want to look somewhat like:(${t_d2434866c44b4e21ba11e723cd7ae14f})

Renaming a File(${t_915206d660fd44468069b7ea1d16b4b4})

Regardless of whether the file is a directory or a document, it can be renamed, with one exception - the root directory has a special name that is expected by the components of a major software vendor's office suite, and the POIFS API will not let that name be changed. Renaming is done by acquiring the file's corresponding Entry instance and calling its renameTo method, passing in the new name.(${t_a2547e9cc0e54beaa29e3c01e417ff55})

Like delete, renameTo returns true if the operation succeeded, otherwise false. Reasons for failure include these:(${t_e14ec79e7d3c4ab292d10df4a82bb4f2})

  • The new name is the same as another file in the same directory. And don't forget - if the new name is longer than 31 characters, it will be silently truncated. In its original length, the new name may have been unique, but truncated to 31 characters, it may not be unique any longer.(${t_dac4f6f0a1594317945fcb377db6b25a})
  • You tried to rename the root directory.(${t_57596193a9d74db08fe1ca0320d6ebdd})
