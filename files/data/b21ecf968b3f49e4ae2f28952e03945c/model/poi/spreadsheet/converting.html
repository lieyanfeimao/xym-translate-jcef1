Upgrading to POI 3.5, including converting existing HSSF Usermodel code to SS Usermodel (for XSSF and HSSF)(${t_bc464e5f71424580b8c5f9a52b4ca875})

Things that have to be changed when upgrading to POI 3.5(${t_f9201cb1fc684fca90ea3f59a05ce428})

Wherever possible, we have tried to ensure that you can use your existing POI code with POI 3.5 without requiring any changes. However, Java doesn't always make that easy, and unfortunately there are a few changes that may be required for some users.(${t_d901aaeaf9d646ee87b673199155d981})

org.apache.poi.hssf.usermodel.HSSFFormulaEvaluator.CellValue(${t_ee8bed696d0045bcb88c67861d4280e0})

Annoyingly, java will not let you access a static inner class via a child of the parent one. So, all references to org.apache.poi.hssf.usermodel.HSSFFormulaEvaluator.CellValue will need to be changed to org.apache.poi.ss.usermodel.FormulaEvaluator.CellValue(${t_467384ac633249ea9ed05595074a6e26})

org.apache.poi.hssf.usermodel.HSSFRow.MissingCellPolicy(${t_ef990f37cf5d418cad4d1539fe9db105})

Annoyingly, java will not let you access a static inner class via a child of the parent one. So, all references to org.apache.poi.hssf.usermodel.HSSFRow.MissingCellPolicy will need to be changed to org.apache.poi.ss.usermodel.Row.MissingCellPolicy(${t_20b3cadb3fd34523868e590bfde27111})

DDF and org.apache.poi.hssf.record.RecordFormatException(${t_48add1437ca84af39e57d13e9c954e77})

Previously, record level errors within DDF would throw an exception from the hssf class hierarchy. Now, record level errors within DDF will throw a more general RecordFormatException, org.apache.poi.util.RecordFormatException(${t_1d0448677caf445eb74b27b84cf54399})

In addition, org.apache.poi.hssf.record.RecordFormatException has been changed to inherit from the new org.apache.poi.util.RecordFormatException, so you may wish to change catches of the hssf version to the new util version.(${t_456f8f78eac24dc29238256d06e7e153})

Converting existing HSSF Usermodel code to SS Usermodel (for XSSF and HSSF)(${t_c84b9dde21de4e7bb0045d17c34936e4})

Why change?(${t_6a380a4b9dfe4e4c951887f8a2107a4f})

If you have existing HSSF usermodel code that works just fine, and you don't want to use the new OOXML XSSF support, then you probably don't need to. Your existing HSSF only code will continue to work just fine.(${t_060b54d207c04840b3167165df3c9723})

However, if you want to be able to work with both HSSF for your .xls files, and also XSSF for .xslx files, then you will need to make some slight tweaks to your code.(${t_172d866be8d949f69529026640207b0a})

org.apache.poi.ss.usermodel(${t_faccfbee246440e28c572f9c468afbbb})

The new SS usermodel (org.apache.poi.ss.usermodel) is very heavily based on the old HSSF usermodel (org.apache.poi.hssf.usermodel). The main difference is that the package name and class names have been tweaked to remove HSSF from them. Otherwise, the new SS Usermodel interfaces should provide the same functionality.(${t_85541e59619c4c70adba062a14104ec2})

Constructors(${t_5865371b5ee24ffd95e6f25dd5286458})

Calling the empty HSSFWorkbook remains as the way to create a new, empty Workbook object. To open an existing Workbook, you should now call WorkbookFactory.create(inp).(${t_20f34178ea944ede9597afca154be7c2})

For all other cases when you would have called a Usermodel constructor, such as 'new HSSFRichTextString()' or 'new HSSFDataFormat', you should instead use a CreationHelper. There's a method on the Workbook to get a CreationHelper, and the CreationHelper will then handle constructing new objects for you.(${t_92e78fadff4d4c63a3801779e6bff194})

Other Code(${t_b35586007c4741ff87f70d7e5b4b2810})

For all other code, generally change a reference from org.apache.poi.hssf.usermodel.HSSFFoo to a reference to org.apache.poi.ss.usermodel.Foo. Method signatures should otherwise remain the same, and it should all then work for both XSSF and HSSF.(${t_40c060c9648b4ebab35d8094e9fb8e60})
